THE PROBLEM

Today's AI coding tools are turbo autocomplete, not a team.

One agent at a time

You drive. The AI types faster. When you close the chat, the work stops and the context evaporates.

No memory between sessions

Every new chat is amnesia. You re-explain the codebase, the conventions, the bug you fixed last Tuesday.

No second pair of eyes

The model that wrote the code also "reviews" it. No specs, no acceptance test, no independent QA.

One platform's quota

Claude throttled? You wait. Cursor stalled? You wait. Each tool is a silo with its own ceiling.

THE IDEA

Treat AI agents like a real engineering team.

Roles. Tickets. Parallel tracks. Code review. Persistent memory. And quality gates the human controls.

📋 Role-based personas

Dev tracks own different modules in parallel. QA agents test independently. PM agents review. Each has its own working memory.

🎫 Tickets, not chats

Work flows through a queue: open → claimed → in_progress → submitted → reviewed → closed. Evidence attached at every step.

🧠 Persistent memory

Decisions, bug findings, conventions, all written to disk. New agents read a briefing and pick up exactly where the last one left off.

THE FEATURE EVERYONE MISSES

Tickets aren't just for QA. They're for any expensive task.

Most "multi-agent" demos pick a model per role and stick with it. OnBoard does something more useful: tickets route work to the cheapest agent that can do it.

The operator's working rule

If you (heavy Opus-class agent) think a task will consume
a lot of tokens or quota but isn't complicated,
PAUSE and ask the user before doing it.
We'll file a ticket for a cheaper agent instead.

Opus identifies the work. Human approves the delegation. A ticket is filed with the spec, target, and required outputs. A cheaper agent (Sonnet, Haiku, Gemini Flash) picks it up via the same claim → submit → review loop the QA tickets use.

Reasoning-heavy

Architecture decisions, debug-the-unknown-cause, multi-file refactors → Opus stays on it

Token-heavy, not hard

Batch renames, log triage, repeated test setup, doc cleanup, file shuffles → ticket → cheaper agent

Execution-only

UI test runs, screenshot capture, evidence collection → QA squad on Antigravity (24/7, $20/mo)

The killer insight: the loop turns multi-model cost arbitrage into a workflow primitive. You stop thinking "should I switch models for this?" mid-task. Opus thinks it, the loop routes it, the cheaper agent picks it up. The same primitive that runs QA also runs cost optimization.

SO IT'S NOT A TODO APP

Eight modules. Real domain logic.

"AI built an app" is dismissible. "AI built a system with FIFO inventory accounting and transactional multi-table writes" is not. Specifics, so the scope isn't underestimated:

FIFO inventory accounting

Sales auto-allocate across production batches by expiry date. Single sale line can split across multiple batches when qty exceeds the oldest batch's remaining stock.

Versioned bill-of-materials

Recipes are versioned. Older versions retained for batches produced under them. Cost-per-unit computed live from RM costs and rolled up to product cost.

Transactional multi-table writes

Production batch creation atomically resolves recipe → pre-checks RM → generates sequenced batch_number → inserts usage manifest → deducts stock. Works around D1's lack of interactive transactions with sequential statements + rollback-on-fail.

Pre-flight preview

Before producing, the app shows green rows (enough RM) and red rows (shortfall) live. Submit blocks if any shortfall. Live cost preview if RM costs set.

"Can-produce" computation

Across all recipes vs current RM stock, compute max-produceable qty per SKU with bottleneck-RM identification. A constrained-resource problem, recomputed on every dashboard load.

Severity-grouped alerts

Info / warning / danger with 24-hour dismiss window and deep-link back to source records (product → stock, RM → adjust, batch → detail).

Role-based access

Viewer accounts auto-hide money fields and block all writes; admin sees everything. Cookie-based session, iOS-PWA-friendly strategy.

Multi-platform pricing + i18n PWA

Four sales channels per product with autofill on sale entry. Mobile-first PWA, stepper-style flows, bilingual UI with Buddhist-calendar date.

THE ORCHESTRATION LAYER

The whole team runs on one MCP server. (OnBoard < v.2.0)

An MCP server I built myself — the shared brain every agent reads from and writes to. Without it, this is just a chat window with a fancy name.

What MCP gives us

MCP (Model Context Protocol) is the open spec that lets any AI client — Claude Desktop, Claude Code, Cursor, Antigravity, Codex — call the same set of tools.

I expose a custom server with 29 MCP tools: agent registry, ticket queue, memory read/write, briefing generator, handoff protocol, ticket review, evidence linker, search across past sessions.

Every agent — regardless of which platform it lives on — connects to the same server and sees the same state.

Why this matters

  • Cross-platform memory — Claude reads what Antigravity wrote
  • Cross-session memory — today's agent reads yesterday's decisions
  • Auditable — every read/write is a tool call you can inspect
  • Vendor-neutral — swap models without re-engineering the loop
  • Local-first — runs on the developer's machine; data stays in the repo

The 29 MCP tools fall into five buckets

Agent lifecycle

OnBoard · join · handoff · checkpoint · get_briefing

Ticket queue

create · claim · submit · review · list · cancel

Persistent memory

write · read · search · pin · links · compact

Project context

bootstrap · status · doctor · context_dirs · token_usage

All open-source. Roughly 2,500 lines of Python. The same server scales from a solo project to a multi-agent team because the abstractions (agents, tickets, memory) are universal. github.com/swisspra/On_Board

A SPECIFIC EXAMPLE

The ORM bug that looked like a UI bug.

Symptom

The product list page showed 0 in stock for every product. Detail pages were correct.

Wrong fixes I almost shipped

  • "It's a stale React Query cache" — wasn't
  • "It's a custom-domain DNS lag class" — wasn't
  • "It's a viewer-role data-strip bug" — wasn't

Real cause

Drizzle's ${productBatches} table interpolation, when used inside a correlated subquery's FROM, was lifting the table reference into the outer query's FROM. Cartesian product. SUM returned 0.

Fix

Replace template-tag references with raw SQL strings inside subqueries. Three lines. Verified in production within 4 minutes.

Why this matters for the pitch

This bug took ~25 active minutes from "user reports" to "shipped fix verified by QA agent."

  • Operator reported the visual bug
  • desktop-opus4.7 hypothesized 3 wrong causes, then queried D1 directly to rule them out
  • Found the real bug class · wrote a memory entry warning future agents
  • Patched · build · deploy · curl-verified · approved

Without persistent memory, the next agent (after handoff) would have wasted hours on the same wrong hypotheses. With memory, the warning is one search away: memory_search "Drizzle subquery"

EIGHT BUGS THE QA AGENT CAUGHT BEFORE THE HUMAN

Not theoretical. Eight real bugs, mid-flight.

Every one of these caused at least one reject → fix → retest cycle. Every one would have shipped in a single-agent workflow because the model that wrote the code would have "reviewed" it and missed.

ORM subquery bug

Product list page summed to 0 across all rows. Drizzle's correlated subquery was lifting the inner table reference into the outer FROM. Cartesian product. 25 minutes from report to deployed fix.

Empty SKU silent fail

Zod schema mismatch swallowed inline errors in product create. UI showed nothing, request 4xx'd silently. QA caught it because the form ran in a real browser.

Trash icon unresponsive

window.confirm blocked behind a re-render. QA tapped, nothing happened. Fix: replaced with a proper Dialog component.

Total-stock card missing

Dashboard spec required a total-stock value summary; first build shipped without it. QA flagged on first render. Added before merge.

Sales row totals missing

Sales history rows didn't show qty + total amount. Worse: when added, viewer-role accounts could see them — needed RBAC gating. QA caught both passes.

Date filter timezone bug

Today's sales excluded for users in non-UTC timezones during early-morning hours. toISOString() defaulted to UTC. QA caught by running the filter at 03:00 local.

Alert severity thresholds

Spec said warn=30d / crit=7d near-expiry. Build shipped 30/90. Off-by-window. QA caught by checking the alert classification against the spec verbatim.

Dashboard money leak

Viewer-role accounts saw aggregate revenue numbers on the dashboard despite RBAC rules. QA logged in as viewer and screenshot-verified the leak.

HONEST GAPS

What today's release doesn't do — all addressable.

No fully autonomous mode

A human is in the loop — as the acceptance tester. By design. The QA agent gates per-merge; the human gates the running app.

No cross-project memory yet

Each .agent-mem/ is per-repo. If you want one crew across 3 codebases, you federate them manually today. Cross-project federation is on the wishlist.

No interactive operator web console

There IS a live observability dashboard (agent grouping, platform colors, orphan ticket detection, idle indicators — since v3.0). An interactive claim/review/dispatch console is on the wishlist.

No public REST API

It's all MCP-based. Other agents can join if they speak MCP, but there's no REST endpoint for "kick off a job from your IDE." An A2A adapter on the wishlist would partially address this.
